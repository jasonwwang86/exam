# 原型优先开发流程草案

## 1. 文档目的

本文档用于补充 `exam` 当前“按模块开发、前后端通过 REST + DTO 协作、通常后端先行”的工作方式，定义在引入 Google Stitch 等原型工具后，团队应如何调整需求澄清、OpenSpec、接口契约、前后端实现与联调流程。

本文档是团队流程草案，不替代具体功能的 OpenSpec proposal、design、specs、tasks。凡是新功能、跨模块变更、API 变更或架构调整，仍应先形成对应的 OpenSpec 产物，再进入实现。

## 2. 适用范围

- 适用于 `exam-web` 与 `exam-service` 同时参与的需求开发
- 适用于管理端页面、考生端页面、跨模块业务页面与聚合页
- 适用于已有明确页面原型、交互稿、线框图或高保真设计稿的需求

不适用范围：

- 纯后端基础设施调整，且无页面或交互影响
- 纯前端样式微调、静态文案修改、资源替换
- 不涉及用户界面和外部行为的局部实现细节调整

## 3. 核心原则

### 3.1 原型先行，但不以原型替代规格

原型负责表达页面结构、信息层级、核心交互与用户路径，不直接等同于：

- 模块划分
- REST 资源边界
- DTO 结构
- 数据库表结构
- 状态机与业务规则

原型是输入，OpenSpec 和 API 契约才是研发落地的正式依据。

### 3.2 需求按场景收口，实现按模块落地

引入原型后，需求分析的最小单位应优先使用“页面/场景”，而不是直接使用代码模块；实现阶段仍保持模块化边界，避免把页面结构直接映射成代码目录结构。

### 3.3 契约先冻结，再并行开发

当页面原型和场景流程明确后，应先冻结核心 REST + DTO 契约，再组织前后端并行开发。默认不再采用“后端全部做完、前端再开始”的串行流程。

### 3.4 前端提出页面数据需求，后端收口服务契约

前端不应被动等待后端先定义接口；后端也不应脱离页面原型独立拍板 DTO。契约应由前端从页面视角提出数据与交互需求，由后端从领域与服务边界视角收口，最终共同评审确认。

### 3.5 场景闭环优先于接口完成

验收标准不应只看某个 Controller、Service 或页面是否已编码完成，而应看用户场景是否从入口、加载、交互、校验、提交到反馈形成闭环。

## 4. 开发流程总览

原型优先模式下，建议采用以下主流程：

1. 原型设计：明确页面结构、核心流程、关键状态与交互入口
2. 场景拆解：从原型中提炼用户场景、页面职责与涉及模块
3. OpenSpec 建模：按场景建立 proposal、design、specs、tasks
4. 契约设计：前后端共同定义 REST 资源、请求/响应 DTO、错误口径
5. Mock 准备：基于已确认契约准备示例响应、假数据或 mock 服务
6. 前后端并行开发：后端优先实现契约，前端基于 mock 和契约推进页面与逻辑
7. 场景联调：以页面闭环和业务规则为中心验证，而不是只做接口通断验证
8. 验收收口：确认 UI、交互、REST 契约、日志、TraceNo、测试和异常分支完整

## 5. 各阶段输出物

### 5.1 原型阶段输出

原型阶段至少应覆盖：

- 页面清单
- 页面主流程
- 页面之间跳转关系
- 关键操作入口
- 列表、表单、详情、弹窗等主要交互形态
- 加载态、空态、失败态、禁用态等关键视觉状态

原型阶段不强求输出接口定义，但至少要能回答：

- 页面展示哪些信息
- 页面上有哪些动作
- 每个动作成功或失败后页面如何反馈

### 5.2 OpenSpec 阶段输出

OpenSpec 应按“场景”而不是仅按“后端模块”来组织变更，且明确标注：

- 变更归属：`exam-web`、`exam-service` 或两者同时涉及
- 影响页面与业务模块
- 涉及的 REST 端点
- 请求/响应 DTO
- 状态流转与校验规则
- 持久化影响
- TDD 适用范围
- 联调与验收项

### 5.3 契约阶段输出

每个场景应形成一份明确的接口清单，至少包含：

- 资源名与 URL
- HTTP Method
- 查询参数、路径参数、请求体
- 响应体
- 空数据口径
- 校验失败口径
- 权限失败口径
- 业务异常口径
- `TraceNo` 透传和返回要求

建议在 OpenSpec design 或补充的接口清单中，附带最小可运行的 JSON 示例。

## 6. 页面、场景与模块的划分规则

## 6.1 继续按模块开发，但不要把页面直接等同于模块

在 `exam` 中，建议维持以下三层概念：

- 业务模块：如考生管理、题库管理、试卷管理、考试计划管理、成绩管理
- 页面/场景：如“考试计划列表页”“创建考试计划流程页”“管理端首页”
- 共享 UI 或能力：如统一壳层、分页表格、筛选栏、权限判断、请求封装

其中：

- 页面/场景是需求和原型的主表达对象
- 业务模块是实现和目录划分的主边界
- 共享 UI 与基础能力应优先沉淀到 `exam-web/src/shared/`

## 6.2 一个页面包含多个模块时的处理规则

当一个页面包含多个模块的信息或操作时，按以下顺序判断：

1. 判断页面的主业务目标是什么
2. 判断哪个模块对页面主流程负责
3. 判断其他模块是在提供候选数据、统计数据还是辅助操作

处理建议如下：

- 若页面只有一个主业务目标，则页面归属主模块，其他模块通过服务调用或共享组件协作
- 若页面是明显的聚合页，如管理端首页、监控总览、工作台，则定义为“场景页/容器页”，不要强行塞入某个单一业务模块
- 若页面跨多个模块且存在复杂流程编排，应先在 OpenSpec 中按场景建模，再拆分前后端任务，不要先按代码目录争归属

## 6.3 页面跨模块时的实现建议

前端建议：

- 页面容器放在主场景所属模块下，或在确属聚合页时放在专门的场景模块下
- 候选数据源、筛选接口、共享组件优先复用已有模块能力
- 不要为了一个跨模块页面，把多个业务模块的页面代码复制到同一文件中

后端建议：

- 仍按领域服务与 REST 资源边界设计接口
- 不因原型上有多个区块，就机械地创建多个后端模块
- 仅在确有聚合查询或性能需求时提供聚合接口

## 7. REST + DTO 契约定义规则

## 7.1 契约归属原则

REST + DTO 契约采用“前端提出页面数据需求，后端收口正式契约，双方共同评审确认”的方式。

职责划分如下：

- 前端负责提出页面需要哪些字段、哪些筛选项、哪些交互动作、哪些状态反馈
- 后端负责判断资源边界、领域约束、字段暴露范围、状态机、事务边界和异常口径
- 双方共同确认最终接口命名、请求 DTO、响应 DTO 和错误语义

因此，契约不属于前端单方，也不属于后端单方，而属于场景级协作产物。

## 7.2 契约设计顺序

建议顺序如下：

1. 前端根据原型整理页面数据清单与动作清单
2. 后端根据领域模型整理资源边界与状态规则
3. 前后端共同评审 REST + DTO 设计
4. 在 OpenSpec design/specs 中冻结契约
5. 基于契约生成 mock 数据并开始并行开发

## 7.3 DTO 设计约束

DTO 应按用途拆分，避免一个 DTO 复用所有场景。至少区分：

- 查询条件 DTO
- 列表响应 DTO
- 详情响应 DTO
- 创建请求 DTO
- 更新请求 DTO
- 状态变更请求 DTO
- 页面聚合响应 DTO

禁止出现以下长期模式：

- 持久化实体直接返回给前端
- 一个 DTO 同时承载新增、编辑、详情、列表
- 仅因前端省事而把多个无关字段堆进一个超大响应体

## 7.4 聚合页的接口策略

原型驱动下，一个页面不必只对应一个接口：

- 简单 CRUD 页面，可采用主资源接口即可
- 复杂聚合页，可由多个接口组合
- 只有在性能、权限收口或交互一致性确有必要时，才增加聚合接口或 BFF 风格接口

判断标准是“是否有明确的场景收益”，而不是“原型看起来像一页”。

## 8. 前后端协作方式调整

## 8.1 从“后端先做完”调整为“契约先冻结，前后端并行”

引入原型后，推荐流程不再是：

`后端设计接口 -> 后端实现完成 -> 前端开始开发`

而应调整为：

`原型确认 -> 场景与契约确认 -> 前端基于 mock 开发 + 后端并行实现 -> 联调收口`

这样做的原因是：

- 页面形态和交互已经由原型定义，前端没有必要完全等待后端
- 后端可以更早获得页面真实数据需求，减少返工
- 联调问题可以集中在字段口径和规则细节，而不是结构性返工

## 8.2 Mock 的引入方式

契约确认后，前端应允许基于以下任一形式推进开发：

- 固定 JSON 示例
- 本地 mock 文件
- mock server
- 暂时的前端假数据适配层

要求是：

- mock 结构必须来源于已冻结契约
- mock 字段命名必须与 DTO 一致
- 联调开始前应清理临时字段和临时口径

## 8.3 联调责任划分

前端联调重点：

- 字段映射正确
- 页面加载态、空态、异常态完整
- 表单校验与后端校验信息衔接一致
- 权限与路由可见性正确

后端联调重点：

- REST 契约与 DTO 实际返回一致
- 状态流转、唯一性校验、事务与异常口径正确
- 请求/响应日志、异常日志、关键业务日志完整
- `TraceNo` 可串联且遵守脱敏约束

## 9. 针对 `exam` 项目的落地建议

## 9.1 OpenSpec 的组织方式

当需求从原型开始时，OpenSpec change 名称建议优先按“场景/能力”命名，而不是按“纯页面名”或“纯后端表名”命名。例如：

- `add-exam-plan-management`
- `add-online-answering-flow`
- `align-admin-dashboard-with-live-features`

不建议直接使用：

- `add-create-page`
- `add-form-fields`
- `add-exam-plan-table-columns`

## 9.2 `exam-web` 的实现边界

前端继续遵守：

- 业务代码按 `src/modules/<module>/pages|components|services|hooks|types` 组织
- 跨模块复用能力放到 `src/shared/`
- 管理端新增业务模块优先接入登录后的统一主页面
- API 请求逻辑集中管理，避免散落在页面组件中

## 9.3 `exam-service` 的实现边界

后端继续遵守：

- 根包名保持 `cn.jack.exam`
- Controller、Service、DTO、Mapper、Entity、Config、Exception、Common 分层清晰
- 请求/响应模型显式定义，不直接暴露持久化实体
- 涉及请求链路时统一透传 `TraceNo`
- 日志遵守脱敏约束

## 9.4 TDD 策略不变，但切入点前移

原型优先不会改变既有 TDD 规则，只改变“何时开始写测试”的上下文：

- `exam-service` 仍默认先写失败测试，再写实现
- `exam-web` 的业务逻辑仍优先先写失败测试，再写实现
- 纯样式和静态排版仍可采用针对性验证而不强制 TDD

变化点在于：

- 测试用例应以已确认的场景和契约为依据
- 前端测试不再只围绕现成后端返回值，而应围绕页面行为和已冻结 DTO

## 10. 常见风险与约束

### 10.1 把原型误当成接口设计

风险：

- 页面有几个区块，就设计几个接口
- 原型上一个操作按钮，就对应一个独立资源

约束：

- 以领域资源边界和场景收益决定接口，不以视觉区块直接切接口

### 10.2 页面跨模块后强行争归属

风险：

- 过早按代码目录争论归属
- 把聚合页硬塞进单一业务模块，导致职责失真

约束：

- 先定义场景职责，再定义模块分工

### 10.3 DTO 过度复用

风险：

- 一个 DTO 兼容列表、详情、新增、编辑
- 后续字段越来越多，前后端都难维护

约束：

- DTO 按场景和用途拆分，保持边界清晰

### 10.4 联调阶段才第一次发现规则

风险：

- 原型只定义外观，没有定义业务规则和异常规则
- 到联调时才讨论字段是否必填、状态能否流转

约束：

- 这些规则必须在 OpenSpec 和契约阶段前置明确

## 11. 需求评审检查清单

一个基于原型的新需求，在进入开发前，至少应完成以下检查：

- 原型是否覆盖主流程、关键状态和核心交互
- 是否已识别页面对应的主场景与涉及模块
- 是否已明确哪些页面是聚合页，哪些是单模块页
- 是否已形成 OpenSpec proposal、design、specs、tasks
- 是否已明确受影响的 `exam-web` 与 `exam-service`
- 是否已冻结关键 REST 端点和 DTO
- 是否已明确校验规则、状态流转、异常口径与权限要求
- 是否已准备 mock 数据或 mock 服务
- 是否已明确前后端联调负责人和验收场景

## 12. 推荐的标准工作流

对于 `exam` 后续采用原型优先方式的需求，推荐统一使用以下工作流：

1. 产品/设计输出原型
2. 研发按原型拆场景，而不是直接拆代码模块
3. 先写 OpenSpec，明确边界、契约、规则和任务
4. 前端提出页面数据与交互需求
5. 后端收口 REST 资源、DTO 和状态规则
6. 双方冻结契约并准备 mock
7. 前后端并行开发
8. 按场景联调与验收
9. 收口测试、日志、TraceNo 和文档
