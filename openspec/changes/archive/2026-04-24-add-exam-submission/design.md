## Context

当前仓库已经具备考生端登录、身份确认、可参加考试列表和在线答题能力，`exam-service` 也已经落地 `exam_answer_session` 与 `exam_answer_record` 两类过程数据模型。但在线答题现阶段只支持过程保存与超时只读，尚未把答题会话收敛为“已提交”的最终状态，也没有手动交卷入口、自动交卷后端兜底或重复提交保护。

“提交试卷”位于路线图中的“考生考试主流程”末端，依赖已完成的 `candidate-login-and-profile-confirmation` 与 `online-answering-flow`，并同时影响 `exam-web` 的考生端交互、`exam-service` 的候选端 REST、会话状态机、数据库迁移与日志策略。由于本次变更横跨现有答题会话模型、前端页面状态切换、自动任务与幂等并发控制，适合单独设计后再进入实现。

## Goals / Non-Goals

**Goals:**
- 在 `exam-web/src/modules/candidate-portal/` 中补齐主动交卷确认交互，并在交卷成功后展示提交结果视图。
- 在 `exam-service` 中提供考生端交卷接口，支持主动交卷、截止自动交卷以及最终状态查询所需的 DTO 契约。
- 基于现有 `exam_answer_session` 持久化最终提交状态、提交时间和提交方式，防止同一考生同一考试重复提交。
- 保证交卷后不能继续保存答案，也不能把已提交考试重新恢复成可编辑答题会话。
- 让服务端在考生页面关闭、断网或前端未发起请求时，仍能在截止时间后完成自动交卷。
- 明确 TDD 范围：`exam-service` 全量适用 TDD；`exam-web` 中交卷确认、结果状态切换、重复触发保护、会话恢复和列表状态展示属于业务逻辑，适用 TDD；纯样式与排版仍通过针对性页面验证完成。
- 在 proposal、design、tasks 与 specs 中持续标注本模块包含“主动提交、到时自动提交、提交状态记录、防重复提交”，不扩展到成绩单、判分或监考大屏。

**Non-Goals:**
- 不重做在线答题的题目加载、答题记录保存或路由结构，只在既有流程上接入交卷闭环。
- 不实现成绩单、客观题判分、主观题阅卷、结果发布或任何成绩查询页面。
- 不实现监考大屏、动态大屏、防作弊、WebSocket 推送或多端同时作答协调。
- 不引入新的独立提交聚合表去复制答题记录，优先在现有答题会话模型内收敛最终提交状态。

## Decisions

### Decision 1: 以 `exam_answer_session` 作为唯一提交状态聚合，而不是新增独立交卷表

本次继续以 `exam_answer_session` 作为同一考生同一考试的唯一过程与终态聚合，扩展其状态枚举和最终提交字段，而不是再新增独立的 `exam_submission` 表。建议在 `exam_answer_session` 中补充 `submitted_at`，并把稳定终态统一收敛为 `SUBMITTED` 与 `AUTO_SUBMITTED` 两类；现有 `TIME_EXPIRED` 不再作为用户可见的最终状态，而是通过迁移或运行时兜底转换为自动交卷终态。

这样做的理由是当前答题会话已经具备 `(exam_plan_id, examinee_id)` 唯一约束，天然满足“一场考试只存在一个最终提交结果”的业务约束。若另建提交表，不但需要维护双向关联和状态一致性，还会增加重复写入、并发更新和回查复杂度。扩展现有会话模型即可同时满足提交状态记录、防重复提交和结果查询。

### Decision 2: 使用单一交卷接口配合共享终态收口服务，保证手动交卷与自动交卷幂等

后端新增 `POST /api/candidate/exams/{planId}/submission` 作为显式交卷接口，由 `cn.jack.exam.controller.candidate` 暴露，`service.candidate` 中新增共享的提交收口服务完成以下职责：
- 校验当前考生、考试计划和答题会话归属。
- 根据当前时间与会话截止时间决定终态是 `SUBMITTED` 还是 `AUTO_SUBMITTED`。
- 通过“仅允许从进行中状态更新为最终状态”的条件更新保证并发幂等。
- 若发现会话已经是 `SUBMITTED` 或 `AUTO_SUBMITTED`，直接返回既有结果摘要而不是再生成第二次提交。

这样可以把“主动交卷”“截止触发的自动交卷”“重复点击交卷按钮”“手动交卷与自动任务并发撞车”全部收敛到一条状态机路径里。备选方案是为自动交卷和手动交卷分别写两套更新逻辑，但那会让重复提交保护难以保持一致。

### Decision 3: 自动交卷采用“前端即时触发 + 服务端定时兜底”的双保险策略

为满足“到时自动提交”，系统需要同时覆盖两种场景：
- 考生仍停留在答题页时，`exam-web` 在倒计时归零后立即触发交卷接口，请求服务端返回自动交卷结果，页面可无缝切换到结果视图。
- 考生已经关闭页面、断网或浏览器冻结时，`exam-service` 通过定时扫描 `deadline_at <= now` 且仍处于进行中状态的会话，调用同一收口服务把会话落为 `AUTO_SUBMITTED`。

仅依赖前端倒计时无法覆盖离线场景，仅依赖定时任务又会造成页面打开场景下的结果反馈延迟。因此本次采用前后端组合方案：前端负责即时反馈体验，后端负责业务正确性和兜底。

### Decision 4: `exam-web` 继续扩展 `candidate-portal`，在现有答题页内切换“作答态”与“交卷结果态”

前端仍在 `exam-web/src/modules/candidate-portal/` 中扩展能力，不新建平行模块。建议继续复用现有答题页路由和状态入口，在页面内根据会话状态切换两种视图：
- `IN_PROGRESS`：展示倒计时、题号导航、题目内容、保存按钮和主动交卷按钮。
- `SUBMITTED` / `AUTO_SUBMITTED`：隐藏可编辑控件，改为展示提交结果卡片，包含提交方式、提交时间、已答/总题数摘要和后续提示。

考试列表页继续作为统一入口页，但其状态文案与按钮策略需要调整：已提交或自动提交的考试必须展示交卷状态摘要，不再允许进入可编辑答题页。这样既能复用既有考生端结构，也能让“刷新页面后继续看到提交结果”落在同一模块中实现。

### Decision 5: 答题会话与考试列表 DTO 同步扩展最终状态字段，保存答案接口在终态后一律拒绝写入

为了让 `exam-web` 在刷新、回退列表、重复点击和自动交卷后都能稳定恢复状态，本次对现有候选端 DTO 做最小必要扩展：
- `GET /api/candidate/exams` 的列表项增加提交终态摘要字段，例如提交状态、提交时间和是否允许继续进入答题。
- `PUT /api/candidate/exams/{planId}/answer-session` 在会话已最终提交时返回终态摘要，供前端直接渲染结果视图，而不是误恢复为可编辑答题会话。
- `PUT /api/candidate/exams/{planId}/questions/{paperQuestionId}/answer` 在会话已最终提交后必须拒绝写入，并返回可读错误提示或带终态的业务响应。
- `POST /api/candidate/exams/{planId}/submission` 返回统一的提交结果响应，供手动交卷与自动交卷共用。

这使前端不需要依赖本地猜测状态，而是始终以服务端终态为准。备选方案是只让交卷接口返回结果，其他接口不暴露最终状态；但那会让刷新恢复和列表状态展示不稳定。

### Decision 6: 后端代码继续放在 `cn.jack.exam` 既有候选端分层中，并为自动交卷补齐可复用日志约束

建议后端放置方式如下：
- `cn.jack.exam.controller.candidate`: 扩展候选端 Controller，新增交卷接口，继续承接答题会话和答案保存接口。
- `cn.jack.exam.service.candidate`: 拆分出交卷收口、自动交卷扫描、答题会话查询与状态拒绝逻辑。
- `cn.jack.exam.dto.candidate`: 新增交卷结果响应，并扩展考试列表项和答题会话响应 DTO。
- `cn.jack.exam.entity`、`mapper`、`resources/mapper`: 扩展 `ExamAnswerSession` 与相关查询 SQL，支持终态字段和自动交卷扫描。
- `config`: 若使用 Spring 定时任务，则放在通用调度配置下，不绑定到单一业务页面。

日志方面，主动交卷成功、自动交卷成功、重复交卷命中、已交卷后保存被拒绝都必须输出与 `TraceNo` 关联的业务日志，并继续遵守“不得输出明文答案、完整 Token、完整 Authorization 头或身份证号”的统一脱敏边界。自动任务触发的交卷日志虽然不一定来自前端请求，也应复用同一套业务日志格式，便于联调与排障。

### Decision 7: 先以失败测试固定状态机和幂等边界，再实现前后端交互

本次变更的核心风险不在页面样式，而在状态机与并发边界。因此实现顺序必须遵守仓库 TDD 约束：
- `exam-service` 先写失败测试，覆盖手动交卷成功、重复交卷幂等、截止后自动交卷、已交卷后拒绝保存、列表状态展示、答题会话刷新恢复终态与日志脱敏。
- `exam-web` 先写失败测试，覆盖交卷确认弹窗、取消交卷不落请求、手动交卷后结果视图、倒计时归零后的自动交卷结果、列表页已提交状态和按钮禁用。
- 样式、文案和排版属于 `exam-web` 非 TDD 豁免部分，通过针对性页面验证确认桌面与移动宽度下可用。

## Risks / Trade-offs

- [把最终提交状态收敛到 `exam_answer_session` 会让同一张表承担过程态与终态] → 通过保持字段最小化、终态枚举清晰和 Mapper 查询职责单一来控制复杂度。
- [前端即时自动交卷与后端定时任务可能并发命中同一会话] → 通过共享收口服务和条件更新保证只有第一次状态迁移生效，后续请求只返回既有结果。
- [把 `TIME_EXPIRED` 迁移为最终提交状态需要考虑历史数据兼容] → 通过数据库迁移脚本或启动时修复，把历史超时会话统一转换为 `AUTO_SUBMITTED` 并补齐 `submitted_at`。
- [考试列表需要同时展示作答态与提交态，状态文案复杂度上升] → 通过明确枚举映射和统一 DTO 字段来源，避免前端在多个页面各自拼接状态。
- [本次不实现成绩单可能让“提交结果展示”看起来较轻] → 这是有意收敛的范围，结果页只承担“确认已交卷”的职责，不承担成绩输出。

## Migration Plan

1. 在 `exam-service` 的 MySQL DDL、测试 schema 与初始化数据中扩展 `exam_answer_session` 的最终提交字段和状态索引，并准备历史 `TIME_EXPIRED` 数据向 `AUTO_SUBMITTED` 的迁移方案。
2. 先编写并跑通 `exam-service` 失败测试，固定交卷状态机、重复提交幂等、自动交卷兜底和日志脱敏边界。
3. 在 `cn.jack.exam` 现有候选端分层中实现交卷接口、自动交卷扫描、答题保存终态拒绝、考试列表状态扩展和答题会话终态恢复。
4. 编写并跑通 `exam-web` 失败测试，再实现 `candidate-portal` 下的交卷确认、结果展示、自动交卷状态切换和列表状态文案。
5. 联调时重点验证：主动交卷、超时自动交卷、刷新后仍显示提交结果、重复点击不生成第二次提交、已交卷后不可继续保存、列表状态一致、`TraceNo` 关联与日志脱敏。
6. 回滚策略以应用层回退为主；若数据库字段已发布，可保留新增字段不继续写入，避免破坏已经落库的交卷状态记录。

## Open Questions

当前没有阻塞 proposal 落地的开放问题。本次默认采用“手动交卷和自动交卷共用一个提交接口收口、自动任务负责离线兜底、结果页只展示提交摘要不展示成绩”的方案；若后续需要成绩发布、交卷后回看答案、提交凭证下载或监考联动，应另起 OpenSpec change 扩展。
