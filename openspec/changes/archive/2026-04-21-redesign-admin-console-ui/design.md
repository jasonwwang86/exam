## Context

`exam-web` 当前已经具备管理员登录、统一主页面入口、管理首页与考生管理模块，但视觉语言仍更像基于默认组件的标准后台皮肤，而不是成熟考试平台的业务工作台。用户已明确选定 `Direction C - Enterprise Exam Platform` 作为目标方向，其特点是浅色、扁平、克制、左侧固定导航、顶部工作区标签页、白底主工作区，以及“首页偏工作台、列表页偏业务表格”的结构风格。

本次变更只影响 `exam-web`。`exam-service` 的接口、DTO、鉴权逻辑、权限模型、数据库结构、`TraceNo` 链路、日志脱敏和 Mapper 设计均保持不变。前后端边界仍然通过现有 REST 契约解耦；`cn.jack.exam` 包下无需新增 Controller、Service、DTO、Mapper 或配置类。

当前已通过 Phase 0 与 Phase 1 文档冻结本次行为边界，并在 Figma 文件中完成方向收敛。接下来的实现目标是将 Figma 中的方向落为 `exam-web` 可复用的共享页面骨架，而不是针对单页做一次性样式微调。

## Goals / Non-Goals

**Goals:**

- 将 `exam-web` 的统一后台壳层重构为浅色扁平的考试业务工作台结构。
- 为管理首页建立更贴近真实业务平台的工作台布局，包括个人信息、统计概览、常用功能和趋势图表区。
- 为列表型业务页建立稳定的三段式结构：筛选区、工具栏、表格区。
- 统一登录页、后台壳层、首页和考生管理页的品牌语言与视觉层级。
- 将布局、页面容器、工作区头部和业务列表页骨架优先沉淀到 `exam-web/src/shared`，便于后续模块扩展复用。
- 在不改变登录、权限、路由和 REST 调用行为的前提下完成前端重设计。
- 对菜单高亮、默认跳转、权限显隐等前端行为继续保留测试保护；纯视觉调整采用构建校验与人工验收。

**Non-Goals:**

- 不新增或修改 `exam-service` 的任何接口、DTO、数据库表结构或 MyBatis-Plus Mapper。
- 不引入新的 UI 框架，不替换 React、Vite、React Router 或 Ant Design。
- 不调整管理员登录流程、退出登录流程、会话恢复规则、默认授权页决策和权限模型。
- 不扩展考生管理的业务能力，不增加新筛选字段或新的业务模块。
- 不在本次变更中引入新的日志链路、TraceNo 处理规则或服务端安全策略变更。

## Decisions

### 1. 采用 `Direction C` 作为唯一设计落地方向

本次设计以 `Direction C - Enterprise Exam Platform` 为唯一实施基准，不再在深色中控台和泛化现代后台之间摇摆。页面视觉将明显向“成熟考试业务系统 / 政企业务后台”收敛，而不是追求概念化的产品后台表现。

选择原因：

- 用户已明确认可该方向最贴近目标参考图。
- 该方向更符合考试业务系统的长期使用场景，重视扫描效率和界面稳定性。
- 在当前 `exam-web` 的模块规模下，这种风格比深色驾驶舱更易于稳定扩展。

备选方案：

- 延续 `Direction B` 的更泛化现代工作台风格：设计感更强，但容易偏离参考图。
- 采用 `Direction A` 深色中控台：气质更强，但不符合当前用户给出的目标系统风格。

### 2. 统一壳层采用“左侧导航 + 顶部工作区标签 + 白底主内容”的三层结构

登录后页面壳层统一采用左侧固定导航、顶部轻量工作区头部和白底主内容区。左侧导航承载模块树状结构，顶部区域用于显示当前工作区标签、账户信息和通用动作，主内容区采用白色工作面板承载首页和列表页内容。

选择原因：

- 这与参考图的交互节奏一致，更接近真实考试管理系统。
- 左侧导航与顶部工作区标签组合更适合后续模块扩展，而不必为每个模块重新定义入口结构。
- 白底主工作区可以减弱背景存在感，提高表格与表单可读性。

备选方案：

- 保持当前“单层顶部 + 侧边栏”壳层：结构较简单，但缺少工作区感。
- 采用更重的卡片式内容容器：实现快，但无法建立成熟业务系统的秩序感。

### 3. 首页采用“工作台”而非“卡片概览页”

管理首页不再只展示简单指标卡和说明文本，而是重构为更像考试业务工作台的结构。首页应至少包含个人信息摘要、当期统计概览、常用功能入口和图表概览区，以强化其作为统一管理入口的定位。

选择原因：

- 参考图明确展示了“个人信息 + 数据概览 + 常用功能 + 趋势图”的工作台布局。
- 这种结构更容易被业务用户理解，也更利于后续继续接入新的模块摘要。
- 比通用 KPI 卡片页更贴近考试系统运营场景。

备选方案：

- 保留现有指标卡 + 说明卡布局：实现成本更低，但业务辨识度不足。
- 直接做成数据大屏：与目标系统风格不匹配，也不利于日常操作。

### 4. 列表型业务页统一采用“筛选区 -> 工具栏 -> 表格区”的三段式结构

考生管理以及后续类似模块页统一使用稳定的三段式结构：顶部筛选区用于输入关键条件，中部工具栏承载新增、导入、导出等操作，底部主表格区作为数据主工作面。空态、错误态和加载态均在表格承载区内统一展示。

选择原因：

- 该结构与参考图中的成熟业务后台一致。
- 对高频业务操作而言，这种结构比多卡片、多层容器更稳健。
- 可以沉淀为 `shared/` 中的复用模式，降低后续模块接入成本。

备选方案：

- 保留更自由的模块页布局：灵活但会继续放大风格漂移。
- 将筛选和工具操作混在一个区块：布局更紧凑，但不利于业务扫描。

### 5. 登录页与后台主界面统一品牌语言，但仍保持聚焦式入口

管理员登录页继续保持未登录场景下的聚焦式单入口结构，不复用登录后壳层；但其颜色、按钮、输入框、边界和字体层级要与统一后台主题保持同一产品语言。页面气质应更偏“企业系统登录入口”，而不是宣传型欢迎页。

选择原因：

- 与现有 `admin-authentication` 规格兼容，不需要改动认证行为。
- 能让整套系统从登录页开始建立统一识别感。
- 避免引入多余的介绍区、营销文案或左侧装饰分栏。

备选方案：

- 完全沿用当前登录页视觉：会破坏新后台的整体统一性。
- 直接复用登录后壳层：会削弱登录页的聚焦性。

### 6. 实现策略以 `exam-web` 共享层重构为核心，TDD 仅覆盖行为逻辑

本次实现的重点是 `exam-web/src/shared` 下的布局、样式与页面骨架能力重构，而不是单页 CSS 微调。菜单高亮、默认跳转、权限态显隐等前端行为仍属于 `exam-web` 的业务逻辑范围，需要保留或补充测试；纯视觉、排版、背景、边框、阴影和空间节奏调整属于 `exam-web` 的非 TDD 豁免项，通过构建与人工验收验证。

选择原因：

- 符合仓库中对 `exam-web` TDD 适用范围的约束。
- 可以避免为低价值的视觉变化编写脆弱测试。
- 保证真正需要保护的行为仍然有自动化兜底。

备选方案：

- 整体不测，只做人工验收：会放大行为回归风险。
- 对所有视觉细节补测试：维护成本高，收益低。

## Risks / Trade-offs

- [Risk] 参考图风格更偏传统业务系统，若实现不当，可能重新退化为普通蓝白后台。 → Mitigation：在实现中优先控制布局层级、边界、留白和工作区结构，而不是简单替换颜色。
- [Risk] 重构共享壳层会同时影响登录后所有模块页面，容易带来滚动、间距和响应式细节问题。 → Mitigation：先重构 `shared` 层，再逐页迁移首页与考生管理页，并在桌面与窄屏分别验收。
- [Risk] 顶部工作区标签页引入后，若仅做视觉占位，可能和当前路由模型脱节。 → Mitigation：先以稳定、轻量的视觉工作区头部承载当前模块上下文，不在本次变更中扩展复杂的多页签状态管理。
- [Risk] 表格与筛选区改造后，现有测试可能因为 DOM 结构变化而失效。 → Mitigation：更新测试为基于用户可见行为和语义查询的断言，减少结构耦合。

## Migration Plan

1. 在 `exam-web` 中基于 `Direction C` 更新全局视觉 token 与基础背景规则。
2. 重构 `exam-web/src/shared/layouts/AdminLayout`，完成左侧导航、顶部工作区和白底主内容区结构。
3. 重构 `exam-web/src/shared/components/admin-page` 或等价共享骨架，建立首页与列表页复用模式。
4. 迁移管理员登录页，使其与新后台主题一致但保持聚焦式登录结构。
5. 迁移管理首页，落地个人信息、统计概览、常用功能与趋势图区。
6. 迁移考生管理页，落地筛选区、工具栏和表格区的三段式布局，同时保留现有接口和权限行为。
7. 运行前端测试和构建，并对登录页、首页、考生管理页进行桌面端与窄屏人工验收。

回滚策略：

- 若新壳层或新页面布局引发严重问题，可直接回滚 `exam-web` 本次变更提交；由于本次不修改 `exam-service`、数据库和接口契约，不存在服务端兼容性回滚成本。

## Open Questions

- 当前阶段不引入新的开放问题。顶部工作区标签页在本次默认作为轻量视觉与上下文承载，不扩展为完整的多标签状态管理能力；若后续确有需求，再单独发起变更。
