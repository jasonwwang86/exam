## Context

当前仓库已经完成管理端登录、考生管理、试卷管理与考试计划管理，但还没有面向考生的独立访问入口。考生端登录与确认信息位于路线图中的“考生考试主流程”起点，依赖现有考生主数据与考试计划数据，需要同时新增 `exam-web` 的考生端页面和 `exam-service` 的认证、身份信息与考试查询接口，并严格把范围收敛在“登录、确认信息、查看可参加考试”，避免提前扩展到在线答题、交卷或成绩单。

## Goals / Non-Goals

**Goals:**
- 在 `exam-web` 中新增落在 `src/modules/candidate-portal/` 的考生端模块，支持考生登录、待考试状态页、身份信息确认与可参加考试列表展示。
- 在 `exam-web` 中为已登录考生补充会话内操作，支持主动刷新可参加考试列表，以及主动退出登录并清理本地缓存。
- 在 `exam-service` 中新增 `cn.jack.exam.controller.candidate`、`service.candidate`、`dto.candidate` 下的考生端认证与查询实现，提供清晰的 REST 契约。
- 复用既有 `examinee`、`exam_plan`、`exam_plan_examinee` 与试卷主数据，不重复建设考生主数据或考试计划存储。
- 复用现有 Bearer Token、`TraceNo`、请求/响应/异常日志与日志脱敏能力，为考生端登录、确认与考试查询增加可检索业务日志。
- 明确本次 change 的 TDD 范围：`exam-service` 全量适用 TDD，`exam-web` 的接口封装、路由守卫、状态切换与确认流程适用 TDD；页面样式与静态展示文案通过针对性验证完成。
- 在 proposal、design、tasks 和 specs 中持续明确包含范围与不包含范围，保证实现不会扩展到在线答题或成绩单。

**Non-Goals:**
- 不实现在线答题、开始考试后的题目加载、自动保存、交卷或成绩查询。
- 不重做管理端登录、权限体系、统一管理台壳层，也不把考生端页面接入管理端主页面。
- 不新增考试结果、答题记录、确认回执持久化等超出本次范围的核心业务表。
- 不新增考生端专用的后端退出登录接口，也不引入超出当前会话范围的服务端缓存失效机制。
- 不实现短信验证码、人脸识别、准考证下载、通知提醒或其他增强型身份校验能力。
- 不定义考试执行态、判分态或成绩态，只处理“登录后待考试”与“可参加考试列表”展示。

## Decisions

### Decision 1: `exam-web` 采用独立的考生端路由分组与模块目录，不复用管理端壳层

`exam-web` 在 `src/modules/candidate-portal/` 下拆分 `pages`、`components`、`services`、`hooks`、`types`，并新增独立的考生端路由分组，例如 `/candidate/login`、`/candidate/confirm`、`/candidate/exams`。该路由分组与管理端 `/login` 和统一管理台壳层隔离，使用针对考生端的轻量布局，避免把考生流程堆入当前 `App.tsx` 的管理端导航逻辑。

这样设计可以保证考生端与管理端在职责、导航和状态管理上边界清晰，同时复用已有 React Router、Axios 和全局样式接入方式。备选方案是让考生入口继续复用管理端壳层或放在同一登录页中切换角色，但这会混淆两套身份与页面结构，并扩大当前 change 的前端改动面。

### Decision 2: 考生端登录使用 `examineeNo + idCardNo` 校验，并签发独立的考生 Token

本次登录方式默认使用考生编号 `examineeNo` 与身份证号 `idCardNo` 作为登录凭据，因为这两项字段已经由考生管理模块维护，且不需要为本次 change 新增密码重置或短信验证能力。`exam-service` 在 `POST /api/candidate/auth/login` 中校验考生存在、未删除、状态为启用且身份证号匹配后，签发独立于管理端的考生 Bearer Token，并在响应中返回考生基础摘要和当前是否已完成本次会话确认。

该方案复用了现有认证基础设施，又能避免把管理端权限模型直接套到考生端。备选方案是引入独立密码、短信验证码或准考证号登录，但这些都要求新增更多主数据或外部依赖，明显超出当前模块边界。

### Decision 3: 考前确认采用会话级确认标记，通过确认接口换发已确认 Token，不新增确认业务表

考生登录成功后默认进入 `exam-web` 的待考试状态页，由 `GET /api/candidate/profile` 返回考生身份摘要、会话确认状态和可读提示。考生在确认页点击确认后，`exam-web` 调用 `POST /api/candidate/profile/confirm`；`exam-service` 校验当前考生会话有效后，换发一个带有 `profileConfirmed=true` 标记的新考生 Token，并返回更新后的会话摘要。`GET /api/candidate/exams` 必须要求已确认会话，未确认时返回受限错误，由 `exam-web` 强制回到确认页。

选择会话级确认而不是持久化确认记录，是因为本次目标只是完成身份信息核对与考试列表可见性控制，还不需要形成可审计的“每场考试确认回执”。备选方案是新增 `exam_plan_confirmation` 或类似表记录确认行为，但这会把边界提前扩展到考试执行审计。

### Decision 4: 可参加考试列表复用考试计划数据，只返回已发布且尚未结束、并明确分配给当前考生的计划

`GET /api/candidate/exams` 基于当前考生 ID 查询 `exam_plan_examinee` 关联关系，并过滤：
- `exam_plan.status = PUBLISHED`
- 当前考生状态仍为启用
- 考试计划未删除，且 `end_time` 晚于当前时间

接口返回的列表项至少包含考试计划 ID、考试名称、试卷名称、开始时间、结束时间、考试时长、当前展示状态和备注摘要，并按开始时间升序排序。列表页可以展示 `待开始` 或 `进行中` 等只读状态，但本次不得提供进入答题、提交试卷或查看成绩的动作入口。

该方案保证考生端只看到“当前仍可参加”的考试，又不强行定义后续在线答题入口。备选方案是把所有历史计划都返回给考生，但那会让“可参加考试列表”语义失真，并把历史查询与成绩查看边界带进本次 change。

### Decision 5: 后端代码按考生端职责单独分层，并在日志中严格脱敏登录凭据和身份字段

后端代码建议放置如下：
- `cn.jack.exam.controller.candidate`：考生登录、身份信息、确认与考试列表接口
- `cn.jack.exam.service.candidate`：登录校验、会话换发、身份摘要拼装、可参加考试查询流程
- `cn.jack.exam.dto.candidate`：登录请求/响应、身份信息响应、确认响应、考试列表响应 DTO
- `cn.jack.exam.mapper` 与 `resources/mapper`：复用或补充考生端所需查询 Mapper/XML
- `config`：扩展考生端鉴权拦截、Token 解析与公开路由配置

日志方面，`exam-service` 必须复用现有 `TraceNo` 透传、请求日志、响应日志、异常日志和脱敏规则，并额外输出考生登录成功/失败、确认完成、考试列表查询的业务日志。日志中不得输出明文身份证号、完整 Token 或完整 Authorization 头，登录失败日志也只能记录脱敏后的考生编号摘要与失败原因分类。

### Decision 6: TDD 先覆盖认证流、路由守卫和过滤规则，再补页面与查询实现

实现顺序遵循仓库约束：
- `exam-service` 必须先写失败测试，覆盖成功登录、登录失败、未确认拦截、确认后查询考试列表、考试过滤规则和日志脱敏。
- `exam-web` 必须先写失败测试，覆盖考生端路由守卫、登录表单校验、未确认重定向、确认后列表展示和空态。
- `exam-web` 的纯样式、布局排版和静态说明文案不强制 TDD，但必须做针对性页面验证，确认登录页、确认页和列表页都已接入样式且不处于裸语义状态。

### Decision 7: 刷新与退出均在 `exam-web` 侧完成，刷新覆盖本地考试缓存，退出清空考生端本地状态

当前考生端已经把 `candidate_token`、`candidate_profile`、`candidate_exams` 缓存在浏览器 `localStorage`。本次在不新增后端接口的前提下，将“刷新考试列表”定义为已确认考生在考试列表页显式触发一次新的 `GET /api/candidate/exams` 请求，用服务端最新返回结果覆盖内存状态与 `candidate_exams` 缓存；失败时保留现有页面并展示错误提示。“退出登录”定义为 `exam-web` 清除 `candidate_token`、`candidate_profile`、`candidate_exams`，重置当前考生端状态并跳转到 `/candidate/login`。

这样可以把增强能力收敛在当前 change 的包含范围内，只增强“登录与确认信息”阶段的会话可用性，不引入新的服务端契约或持久化影响。备选方案是新增 `POST /api/candidate/auth/logout` 或统一的“强制忽略缓存并重新拉取全部会话数据”机制，但这些都超出了本次所需的最小改动面。

## Risks / Trade-offs

- [使用 `idCardNo` 作为登录凭据会提高敏感信息暴露风险] → 通过 HTTPS 前提、输入脱敏展示、日志彻底脱敏和错误提示模糊化降低风险；更强身份认证能力留到后续 change。
- [会话级确认不落库，无法形成每场考试确认回执] → 本次明确只做列表可见性控制，不承担审计职责；若后续需要考试前确认留痕，再通过新的 OpenSpec change 增补。
- [考试列表过滤过严可能让考生看不到已分配但尚未发布的考试] → 当前能力定义为“可参加考试”，因此只返回 `PUBLISHED` 且未结束的计划，未发布计划不应对考生可见。
- [考生端与管理端共用认证基础设施，容易出现鉴权串线] → 通过独立路由前缀、独立 Token claims 和独立 controller/service 分层隔离候选实现。
- [未来开始答题能力接入时，列表页可能需要增加入口动作] → 本次只定义只读列表和待考试状态，后续新增开始答题能力时通过新的 change 增量扩展，不在本次文档中预埋复杂执行逻辑。
- [考试列表缓存可能导致考生看到旧数据] → 通过显式“刷新考试列表”按钮让考生主动拉取最新可参加考试，并在确认前继续禁止访问考试列表。

## Migration Plan

1. 在 `exam-service` 先补考生端认证与考试查询失败测试，确认复用现有考生与考试计划数据即可覆盖本次需求。
2. 扩展考生端 Token、鉴权配置、DTO、Mapper 和 Service，实现登录、身份查询、确认换发与考试列表接口，同时补充 TraceNo 关联日志与脱敏校验。
3. 在 `exam-web` 新增 `candidate-portal` 模块、独立路由和页面骨架，先完成登录态管理与路由守卫，再接入确认页、考试列表以及刷新/退出会话操作。
4. 联调阶段重点验证：错误凭据登录、禁用考生登录拒绝、未确认无法查看考试列表、确认后能看到已发布且未结束的考试、空列表提示、显式刷新后考试列表覆盖缓存、退出登录后本地状态清空、TraceNo 透传与日志脱敏。
5. 回滚策略以“下线考生端路由入口并回退应用版本”为主；由于本次默认不新增核心业务表，数据库层回滚主要限于鉴权配置或查询层调整。

## Open Questions

当前没有阻塞 proposal 落地的开放问题。本次默认考生使用 `examineeNo + idCardNo` 登录，确认行为只作用于当前考生会话，不形成持久化确认回执；如后续需要独立密码体系、短信验证码、每场考试确认记录或开始考试入口，应通过新的 OpenSpec change 单独扩展，而不是在本次 change 中继续扩边界。
