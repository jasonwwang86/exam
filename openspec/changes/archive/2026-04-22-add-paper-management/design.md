## Context

当前仓库已经完成管理端登录、权限控制、统一管理台壳层、首页、考生管理与题库管理模块，但还没有可复用的试卷实体与手工组卷流程。试卷管理位于路线图的“考务数据建设”阶段，依赖现有题库管理提供的试题主数据，需要同时新增 `exam-web` 的试卷管理页面与 `exam-service` 的 REST、DTO、Mapper 和数据库设计，并严格把范围收敛在“管理端试卷组织与维护”，避免提前扩展到考试计划、在线答题、自动组卷或判分流程。

## Goals / Non-Goals

**Goals:**
- 在 `exam-web` 中新增落在 `src/modules/paper-management/` 的试卷管理模块，支持试卷分页查询、新增、编辑、删除、详情查看和手工组卷。
- 在 `exam-web` 中提供试卷基础信息维护与题目明细维护交互，并继续接入统一管理台壳层，而不是为单一模块重复建设独立首页框架。
- 在 `exam-service` 中新增 `cn.jack.exam.controller.admin`、`service`、`dto`、`mapper`、`entity` 分层下的试卷管理实现，提供清晰的 REST 契约。
- 为试卷与试卷题目明细建立可演进的 MySQL 表结构、MyBatis-Plus Mapper 与初始化/测试数据，支持后续考试计划复用。
- 通过题目明细分值汇总维护试卷总分，并由管理员显式维护试卷时长。
- 复用现有权限、`TraceNo`、请求/响应/异常日志与脱敏规则，为试卷创建、编辑、删除、加题、调分、移除和排序补充可检索的关键业务日志。
- 明确本次 change 的 TDD 范围：`exam-service` 全量适用 TDD，`exam-web` 的接口封装、选题状态、分值汇总、表单校验、权限/路由逻辑适用 TDD；纯样式与布局微调通过针对性验证完成。

**Non-Goals:**
- 不实现题库管理能力本身、考试计划、在线答题、阅卷判分或成绩计算流程。
- 不实现自动组卷、随机抽题、题型配比规则、知识点策略或其他组卷算法。
- 不重做管理端登录、角色权限基础设施、统一管理台壳层或题库管理模块本身。
- 不引入发布态、归档态、版本比对、导入导出或其他超出“试卷基础维护”的生命周期能力。

## Decisions

### Decision 1: `exam-web` 使用单一路由页面承载试卷列表，试卷基础信息与题目明细通过模块内详情工作流维护

`exam-web` 在 `src/modules/paper-management/` 下拆分 `pages`、`components`、`services`、`types`。试卷管理主页面继续沿用统一管理台壳层，以查询区、工具栏和数据表格区承载试卷列表；新建或编辑试卷时，先维护基础信息，再在同一模块内进入题目明细工作流进行手工组卷。题目明细维护通过详情抽屉或详情页内的局部工作区承载，而不是新建独立首页级框架。

这样可以复用现有 `AdminLayout` 与列表型业务页结构，减少路由和导航复杂度，也便于后续把考试计划继续接入同一管理台。备选方案是为试卷基础信息和组卷明细拆成两个一级路由页面，但会增加状态同步成本，并放大当前模块的导航复杂度。

### Decision 2: REST 契约按 `papers` 与 `papers/{paperId}/questions` 两类资源拆分，选题候选列表复用题库查询能力

后端统一暴露以下受保护资源接口：
- `/api/admin/papers`：分页查询、详情获取、创建、更新、删除
- `/api/admin/papers/{paperId}/questions`：题目明细列表、新增
- `/api/admin/papers/{paperId}/questions/{paperQuestionId}`：题目明细更新、删除

试卷管理中的“从题库选题”不新增独立的候选题资源，而是复用题库管理已提供的 `GET /api/admin/questions` 只读查询能力，由 `exam-web` 在组选题弹层中使用适配后的筛选条件和列表字段。这种设计可以避免在 `exam-service` 中重复维护候选题查询逻辑，同时保持试卷管理与题库管理的边界清晰：前者负责“组织”，后者负责“供题”。

备选方案是新增专门的候选题查询接口，但会重复维护与题库列表高度重合的筛选、分页和字段映射逻辑，收益有限。

### Decision 3: `exam-service` 使用独立试卷表与试卷题目明细表，题目明细保存题库引用与必要快照

后端新增两张核心表：
- `paper`：`id`、`name`、`description`、`duration_minutes`、`total_score`、`remark`、`deleted`、`created_at`、`updated_at`
- `paper_question`：`id`、`paper_id`、`question_id`、`question_stem_snapshot`、`question_type_name_snapshot`、`difficulty_snapshot`、`item_score`、`display_order`、`deleted`、`created_at`、`updated_at`

其中：
- `paper.name` 需要唯一约束，删除采用逻辑删除。
- `paper.duration_minutes` 使用正整数，单位为分钟。
- `paper.total_score` 使用 `DECIMAL(8,2)` 持久化，并由服务端依据当前有效题目明细的 `item_score` 自动汇总维护。
- `paper_question` 使用 `question_id` 关联题库试题，同时持久化题干、题型名称、难度等快照字段，避免题库后续修改直接改变已组好的试卷展示结果。
- 同一试卷下不允许重复添加同一个有效题目；排序由 `display_order` 维护。

后端代码放置建议如下：
- `cn.jack.exam.controller.admin`：`AdminPaperController`、`AdminPaperQuestionController`
- `cn.jack.exam.service.paper`：试卷查询与维护、题目明细维护、总分汇总
- `cn.jack.exam.dto.paper`：试卷与题目明细 DTO
- `cn.jack.exam.entity`：`Paper`、`PaperQuestion`
- `cn.jack.exam.mapper` 与 `resources/mapper`：MyBatis-Plus Mapper 与 SQL
- `resources/db/mysql` 与测试库 schema/data：表结构、索引、权限种子与测试数据

备选方案是只在 `paper_question` 中保存 `question_id` 而不保存快照，但那会让题库内容变更直接影响既有试卷明细，削弱试卷作为稳定编排结果的价值。

### Decision 4: 手工组卷允许按试卷维度维护题目顺序与题目分值，总分由服务端实时汇总而不是单独自由录入

管理员在试卷明细中从题库选择题目后，系统将题库默认分值带入 `paper_question.item_score`，并允许在试卷内单独调整该题目的试卷分值与显示顺序。每次新增、编辑、删除或重排题目明细后，`exam-service` 都必须重新汇总当前试卷有效题目的 `item_score` 并回写 `paper.total_score`，由 `exam-web` 在列表页、详情页和基础信息区统一展示。

这种设计能够同时满足“总分配置”与“题目明细维护”的范围要求，又避免管理员手工填写一个与题目明细不一致的总分值。备选方案是让管理员独立录入总分并要求与题目明细汇总一致，但在当前没有发布态或草稿校验流程的前提下，容易引入不一致状态。

### Decision 5: 试卷基础信息允许先保存空试卷，后续通过手工组卷逐步补全题目明细

本次不引入发布态或考试计划，因此试卷基础信息保存时允许题目明细为空，此时 `total_score` 为 `0`。管理员可以先建立试卷名称、说明、时长和备注，再进入题目明细工作流逐步补全题目，系统在每次明细变更后刷新总分。这种顺序与手工组卷的实际操作更吻合，也能降低表单复杂度。

备选方案是强制“试卷创建时必须至少含一题”，但那会要求在一次交互内同时完成基础信息和组选题，显著提升 UI 和事务复杂度。

### Decision 6: 复用现有 TraceNo 与日志脱敏能力，并把试卷业务日志限定在通用基础能力之上

所有试卷管理接口继续复用现有 `TraceNo` 透传、请求日志、响应日志、异常日志和 `LogSanitizer` 脱敏规则。除此之外，`exam-service` 需要为试卷创建、编辑、删除、加题、调分、移除和排序输出关键业务日志，日志中仅记录试卷 ID、题目 ID、操作类型、结果摘要和脱敏后的字段摘要，不输出完整题干正文或其他敏感内容。

这样既遵守已有日志约束，也能为后续联调、排障和权限审计提供稳定链路。备选方案是只依赖通用 HTTP 日志，但那不足以支撑题目重复添加拒绝、总分汇总异常或排序调整问题定位。

## Risks / Trade-offs

- [题目快照会引入冗余字段和同步成本] → 通过把快照写入限定在“加题”与“更新题目明细”两个入口，并保持字段最小化来控制复杂度。
- [复用题库查询接口会让试卷管理在读路径上依赖题库模块 DTO] → 通过 `exam-web` 单独定义试卷组选题适配类型，避免页面层直接耦合题库页面组件。
- [总分由服务端汇总后，管理员无法手工录入一个独立总分] → 在规格中明确“总分配置”通过题目明细分值维护完成，避免与题目级分值出现双重真相源。
- [允许空试卷保存，可能产生暂不完整的数据] → 通过列表与详情明确展示题目数量和总分，并把考试计划发布约束留到后续独立 change 处理。
- [同一试卷禁止重复加题可能限制某些非常规场景] → 当前考试系统以题库题目唯一编排为默认规则；如未来需要允许重复题，应通过新 change 重新定义试卷明细模型。

## Migration Plan

1. 在 `exam-service` 新增数据库迁移脚本，创建试卷表、试卷题目明细表、索引与菜单/API 权限种子数据，并同步更新测试用 schema/data。
2. 先完成 `exam-service` 的失败测试、DTO、Mapper、Service、Controller、总分汇总与题目快照逻辑，再提供可联调的 REST 接口。
3. 在 `exam-web` 接入试卷管理模块、菜单与路由，并基于接口完成试卷列表、基础信息表单、题目明细列表和手工组选题交互。
4. 联调阶段重点验证：空试卷保存、加题/删题/调分/排序后的总分同步、题目重复添加拒绝、无权限账号行为、`TraceNo` 透传和日志脱敏。
5. 回滚策略以“禁用菜单/权限入口并回退应用版本”为主；若后续模块尚未依赖试卷数据，可通过独立迁移脚本下线试卷相关表与权限数据。

## Open Questions

当前没有阻塞 proposal 落地的开放问题。本次默认试卷基础字段为名称、说明、时长、备注与总分汇总结果；如后续需要发布状态、试卷版本、自动组卷规则、题目分类统计或考试计划关联约束，应通过新的 OpenSpec change 增补，而不是在本次 change 中扩边界。
