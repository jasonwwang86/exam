## Context

当前 `exam-web` 已具备管理员登录、会话恢复、统一主页面入口和考生管理模块，但登录页与登录后的壳层分别维护不同的视觉语言，缺少统一的企业级管理台样式规范。项目依赖中已安装 `antd`，但尚未将其作为共享布局、表单、表格和反馈组件的基础体系使用。

本次变更只影响 `exam-web`。`exam-service` 的鉴权、权限和考生管理接口继续保持现有 REST 契约，不新增端点、不修改请求/响应 DTO，也不涉及 `cn.jack.exam` 包下任何 Controller、Service、Mapper 或数据库结构。`TraceNo`、日志脱敏和后端请求日志链路在本次变更中不做调整。

## Goals / Non-Goals

**Goals:**
- 为 `exam-web` 登录后的统一主页面建立稳定的企业级管理台壳层，统一头部、侧边导航、内容容器和当前菜单高亮方式。
- 让 `exam-web` 的管理员登录页与登录后主界面保持同一主题语言，同时采用去侧栏的居中单卡片登录结构。
- 为首页和列表型业务页面建立可复用的共享页面模式，减少后续模块接入时重复建设页面骨架和样式。
- 在不改变鉴权、权限和现有 REST 行为的前提下，让管理首页与考生管理页面更接近通用企业级管理台的视觉结构和交互节奏。
- 去掉各模块顶部介绍型 Hero，只保留必要标题、操作和内容卡片。
- 将考生管理的新增、编辑、导入统一为弹框交互，并由顶部 `新增` 下拉入口承载“新增考生”和“导入文件”两类动作。
- 明确本次变更的 TDD 范围：纯样式与静态布局调整属于 `exam-web` 的非 TDD 豁免项；若引入菜单选中、路由映射等前端行为逻辑，应补充对应测试。

**Non-Goals:**
- 不改变登录流程、会话恢复和退出登录的交互契约。
- 不新增管理模块，不扩展考生管理的业务能力，不调整筛选字段、权限模型或导入导出规则。
- 不修改 `exam-service` 中任何接口、日志策略、DTO、数据库表结构或 MyBatis-Plus Mapper。
- 不引入 Ant Design Pro、微前端或新的状态管理方案。

## Decisions

### 1. 使用 Ant Design 作为管理台基础组件体系

选择在 `exam-web` 中正式采用 Ant Design 的 `Layout`、`Menu`、`Typography`、`Card`、`Form`、`Input`、`Select`、`Button`、`Table`、`Tag` 等基础组件来承载本次视觉升级，并通过共享主题配置统一颜色、圆角、间距和基础阴影。

选择原因：
- 依赖已存在，符合仓库“优先采用主流、稳定、广泛使用的框架和库”的约束。
- 对管理台场景的导航、表单、表格和反馈组件支持成熟，可减少自定义壳层样式的维护成本。
- 后续新增模块可直接沿用同一套共享组件和样式语义，而不是继续在各模块内部复制 CSS 规则。

备选方案：
- 保留现有纯 CSS Modules 并只重写样式：短期改动更少，但难以形成稳定的共享组件体系。
- 直接引入 Ant Design Pro：能力更完整，但当前项目规模较小，会引入不必要的结构复杂度。

### 2. 将登录后壳层与页面容器沉淀到 `exam-web/src/shared`

`AdminLayout` 保留为统一入口，但内部将升级为更标准的管理台壳层，并将页面标题区、筛选区、操作区、内容容器等可复用结构沉淀到 `exam-web/src/shared`。业务模块页面只负责自身字段、数据渲染和权限控制，不再自行定义整套页面骨架。

选择原因：
- 符合 AGENTS 中“跨模块 UI 优先沉淀到 `shared/`”的约束。
- 后续新增模块接入统一主页面时，可以直接复用页面结构和样式规则。
- 避免 `App.tsx` 或单个业务页面承担过多布局职责。

备选方案：
- 在每个模块页内部各自拼装页面头部和查询区：实现快，但会继续放大跨模块不一致。
- 将全部布局逻辑堆到 `App.tsx`：会让路由入口和布局细节耦合，不利于演进。

### 2.5 登录页与主界面主题一致，但不复用登录后壳层

管理员登录页采用与主界面一致的主题 token、品牌标识、按钮语义、输入框风格和背景氛围，但仍保持单卡片、单入口的聚焦式结构，不引入登录后的侧边导航或顶部工作台头部，也不保留额外的左侧介绍区块。

选择原因：
- 用户明确要求登录页主题与主界面一致。
- 登录页与主界面保持同色系和品牌语言，可以强化系统的一致性。
- 保持独立登录入口结构，可以避免未登录页面看起来像“被裁剪过的后台”。

备选方案：
- 维持登录页当前独立风格：实现更少，但会继续破坏整套系统的视觉统一性。
- 直接复用登录后壳层：视觉最接近，但会削弱登录页的聚焦感并引入无意义的导航框架。

### 3. 模块页采用“紧凑标题 + 内容卡片”而非介绍型抬头

管理首页、考生管理等模块页不再使用带说明文案的顶部 Hero。页面入口只保留必要标题，正文直接进入指标卡、筛选区、操作区和数据区。共享页面组件继续负责统一卡片容器、标题层级和区块间距，但不强制每个模块展示介绍性文案。

选择原因：
- 用户明确要求每个模块不再展示抬头介绍。
- 对管理台高频使用场景而言，去掉介绍型抬头可以压缩首屏高度，让用户更快进入操作区。
- 共享页面规范仍然存在，只是从“Hero + Section”收敛为“Title/Section + Content”。

备选方案：
- 保留 Hero 但隐藏描述：仍会占据较大的首屏空间，和用户目标不一致。
- 完全去掉页面标题：会削弱模块辨识度，不利于多模块扩展。

### 4. 考生管理操作改为弹框流，并通过 `新增` 聚合入口承载

考生管理页面保留查询区和列表区，但不再以内联表单或独立导入面板承载新增、编辑、导入。页面顶部只保留一个 `新增` 主入口，点击后展开下拉菜单，提供 `新增考生` 与 `导入文件` 两个动作；新增、编辑统一使用表单弹框，导入使用文件选择弹框。

选择原因：
- 这是企业管理台中更常见的列表页操作组织方式。
- 将新增和导入聚合到一个主入口，可减少工具栏按钮密度。
- 弹框更适合短流程维护，不会打断当前列表上下文。

备选方案：
- 保持两个独立按钮：实现更直接，但入口分散。
- 改为抽屉：适合更长表单，但当前字段规模不大，弹框更紧凑。

### 5. 保持前后端边界不变，仅重构 `exam-web` 的展示层

本次变更不调整 `exam-web` 与 `exam-service` 的 REST 契约。登录后仍通过现有用户信息中的 `menus` 和 `permissions` 构造导航；考生管理仍调用现有查询、新增、编辑、删除、状态切换、导入和导出接口；前端仅重构布局、视觉层级和控件呈现方式。

选择原因：
- 用户当前目标是统一前端管理台风格，而非调整业务能力。
- 保持接口和数据契约不变可以降低联调风险，并让本次变更聚焦在共享 UI 能力建设。
- 现有 `exam-service` 已满足需求，不需要在 `cn.jack.exam` 下增加新的 Controller、Service、DTO、Mapper 或日志能力。

备选方案：
- 同步重整接口返回结构以匹配新组件：收益有限，但会让前后端一起变动，扩大范围。

### 6. 验证策略区分“行为逻辑”和“纯视觉改造”

对于 `exam-web` 中菜单高亮、路由匹配、权限态切换等可测试行为，继续使用现有前端测试补充或调整断言；对于纯样式、间距、颜色和组件替换，采用构建校验加人工验收，不强制以 TDD 驱动。

选择原因：
- 符合仓库对 `exam-web` 的 TDD 约束边界。
- 能避免将纯展示改动包装成低价值测试，同时保留对关键行为的保护。

## Risks / Trade-offs

- [Risk] Ant Design 组件替换会改变现有 DOM 结构，可能导致已有测试断言失效。 → Mitigation：优先以用户可见行为和可访问性查询更新测试，避免脆弱的结构性断言。
- [Risk] 统一壳层和页面容器同步改动首页、考生页和登录页，容易出现间距、滚动和响应式细节不一致。 → Mitigation：先定义共享主题，再按页面逐个迁移，并在桌面端和窄屏场景做验收。
- [Risk] 使用组件库后若主题 token 约束不清，后续模块仍可能写出风格漂移页面。 → Mitigation：将主题配置、共享容器和常用页面片段集中到 `shared/`，减少模块内随意定义样式的空间。
- [Risk] 过度追求截图复刻会让实现偏离项目实际信息结构。 → Mitigation：只参考企业级管理台的布局语言和视觉层级，不逐像素复刻外部样式。

## Migration Plan

1. 在 `exam-web` 中接入 Ant Design 的全局主题和基础 reset，确保现有路由与鉴权流程不受影响。
2. 重构共享 `AdminLayout`，完成统一头部、侧边导航、主内容容器和当前菜单高亮逻辑。
3. 在 `exam-web/src/shared` 建立标准化页面结构组件或样式约定，再迁移管理首页与考生管理页面。
4. 重构管理员登录页，使其与统一管理台主题一致，但保持独立入口布局。
5. 运行前端测试与构建，确认现有权限和数据链路保持稳定，并对关键页面做视觉验收。

回滚策略：
- 若新壳层或组件替换引发严重样式或交互问题，可回滚 `exam-web` 本次提交；由于未修改 `exam-service` 或数据库，不存在服务端兼容性回滚成本。

## Open Questions
