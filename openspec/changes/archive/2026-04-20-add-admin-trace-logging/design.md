## Context

当前 `exam-service` 已经具备基础认证、权限拦截和异常处理能力，但日志仍停留在“无统一入口、无链路标识、无关键节点记录”的状态。出现登录失败、权限拒绝、接口报错或联调异常时，无法通过同一个标识快速串联 `exam-web` 发起请求、`exam-service` 认证拦截、控制器处理、服务逻辑和异常返回过程。

本次变更同时影响 `exam-web` 与 `exam-service`，属于跨前后端的可观测性增强，主要约束如下：

- 日志增强需要先在当前已有 `exam-web` → `exam-service` 链路中落地，但设计应保持通用复用能力，不绑定到单一业务模块
- 需要保留前后端分离边界：`exam-web` 负责透传 TraceNo，`exam-service` 负责日志采集、脱敏、输出和回写响应头
- 不引入新的数据库表、外部日志平台、分布式链路追踪系统或额外中间件
- 需要新增明确的日志输出约束，避免后续继续输出明文密码、完整 Token 或噪音日志

## Goals / Non-Goals

**Goals:**

- 为 `exam-web` 管理端请求增加统一的 `TraceNo` 请求头透传能力，并确保格式为 32 位无连字符 UUID
- 为 `exam-service` 增加统一的 TraceNo 上下文能力，确保同一次请求的全部日志都能带上相同 TraceNo
- 为 `exam-service` 增加统一的请求开始、请求结束、异常、关键逻辑日志
- 为 `exam-service` 增加日志输出约束，包括日志级别、字段要求、脱敏规则与禁止输出内容
- 明确本次包含范围：TraceNo 透传、请求/响应报文日志、关键逻辑日志、异常日志、日志约束，并作为通用基础设施复用

**Non-Goals:**

- 不接入 ELK、SkyWalking、Zipkin、OpenTelemetry 等外部日志或链路追踪平台
- 不实现跨微服务、跨消息队列、跨第三方系统的分布式 Trace
- 不实现复杂审计日志、操作留痕平台或日志检索后台
- 不变更现有认证、权限、菜单的业务语义和数据库结构

## Decisions

### 1. Trace 标识统一使用 `TraceNo` 请求头和 MDC 上下文

**Decision**

`exam-web` 对管理端 API 请求统一附加 `TraceNo` 请求头，格式为 32 位无连字符 UUID；`exam-service` 在请求进入时优先读取该请求头，若缺失或格式非法则自动生成新的 32 位无连字符 UUID，并写入 MDC 与响应头。后端所有链路日志统一输出 `TraceNo` 字段。

**Rationale**

- 与用户要求保持一致，直接使用 `TraceNo` 作为前后端对齐标识
- 统一成 32 位无连字符 UUID，便于日志检索、长度控制与跨端传递
- 单体 Spring Boot 应用中，MDC 是实现同一次请求日志串联的最小可行方案
- 响应头回写可以让前端或测试工具在缺少请求头时也拿到最终 TraceNo

**Alternatives considered**

- 仅后端生成 TraceNo：实现更简单，但无法满足“从前端 header 到后端完整处理流程”的要求
- 使用其他头名如 `X-Trace-Id`：更常见，但会偏离当前需求约定，增加沟通成本

### 2. 后端使用统一 Servlet Filter 记录请求/响应主日志，而不是分散在控制器中手写

**Decision**

在 `exam-service` 的 `cn.jack.exam.config` 下新增统一日志过滤器，负责：

- 包装请求与响应，捕获 body 摘要
- 记录请求进入日志与请求完成日志
- 计算耗时并输出 HTTP 方法、URI、状态码、客户端 IP、TraceNo
- 在 finally 中清理 MDC，避免线程复用造成串日志

控制器和服务层不负责基础报文日志，只补充关键业务节点日志。

**Rationale**

- 统一过滤器能保证所有 HTTP 接口都经过同一套日志规则
- 避免把重复的日志样板代码散落到控制器和服务方法里
- 与现有 `AdminAuthInterceptor` 形成分层：Filter 负责日志与上下文，Interceptor 负责认证与权限

**Alternatives considered**

- 在拦截器中记录日志：可以实现，但对请求/响应 body 捕获不如 Filter 合适
- 在每个控制器里手写日志：实现快，但难以统一格式，也容易遗漏

### 3. 报文日志只记录可控摘要，并对敏感字段做强制脱敏

**Decision**

请求/响应日志采用“结构化文本 + 脱敏摘要”策略：

- 默认记录方法、路径、查询串、状态码、耗时、TraceNo、用户标识摘要
- 对 JSON 和普通文本 body 记录摘要内容
- 对 `password`、`Authorization`、`token`、`passwordHash` 等敏感字段做脱敏
- 对二进制、大体积、不可安全序列化的 body 不记录原文，仅输出占位说明

**Rationale**

- 既满足排障需要，又避免把敏感信息直接写入日志
- 当前管理端首期以 JSON 接口为主，摘要日志已经能覆盖主要问题定位场景
- 用统一脱敏规则比依赖调用方自觉规避更可靠

**Alternatives considered**

- 完全不记录 body：安全但定位效率不足
- 全量记录所有 body：信息最全，但敏感信息泄露风险和日志噪音都过高

### 4. 关键逻辑日志能力设计为通用规范，当前先在已存在链路落地

**Decision**

`exam-service` 的关键逻辑日志能力设计为通用规范，当前先在已存在链路落地。首批在以下关键节点补充业务日志：

- 登录成功、登录失败、禁用账号拒绝
- Token 鉴权失败、会话过期、权限校验失败
- 退出成功

日志需包含 TraceNo、用户名或用户 ID 摘要、目标接口或权限码、处理结果，但不得输出明文密码或完整 Token。

**Rationale**

- 这些节点直接对应当前系统中最需要排查的实际链路
- 先把现有链路打通，再让后续模块复用同一规范，比为每个模块分别发明日志规则更稳妥
- 日志基础设施本身保持通用，避免绑定到单一模块

**Alternatives considered**

- 覆盖所有 service 方法：日志量过大，且会把首版实现复杂度拉高
- 仅保留请求/响应日志：定位权限与认证问题时信息不足

### 5. 日志输出约束作为本次 change 的显式规范，而不是仅靠代码风格约定

**Decision**

在 OpenSpec 中新增明确的日志约束，至少包括：

- 每条后端链路日志必须包含 `TraceNo`
- `TraceNo` 必须为 32 位无连字符 UUID
- 请求完成日志必须包含方法、路径、状态码、耗时
- 异常日志必须包含异常类型、错误消息和 TraceNo
- 禁止输出明文密码、完整 Token、完整 `Authorization` 头和密码摘要
- 关键业务日志必须使用可检索的稳定文案，避免随意打印

**Rationale**

- 只有把规则写进 change，后续实现和 review 才有统一判断标准
- 约束先行可以避免“先加日志，后面再清理”的返工模式

**Alternatives considered**

- 只在实现时口头约定：容易在后续迭代中失效
- 一开始就引入完整日志规范体系：超出当前最小必要范围

## Risks / Trade-offs

- [请求/响应日志增加性能开销] → 采用通用过滤器但以摘要输出为主，避免全量大报文记录
- [脱敏规则不完整导致敏感信息泄露] → 首期将敏感字段集中封装为统一规则，并通过测试覆盖常见字段
- [前端与后端使用不同 TraceNo 生成策略导致链路不一致] → 统一要求以后端收到的 `TraceNo` 为最终链路标识，并回写响应头
- [日志过多降低可读性] → 基础日志固定为请求开始/结束/异常，业务日志仅覆盖认证与权限关键节点

## Migration Plan

1. 在 OpenSpec 中定义日志与 TraceNo 相关 proposal、design、specs、tasks
2. 在 `exam-web` 中为管理端 Axios 客户端增加 `TraceNo` 请求头透传
3. 在 `exam-service` 中增加统一日志过滤器、TraceNo 上下文与日志脱敏工具
4. 在 `exam-service` 当前已有业务链路中补充关键业务日志，并作为后续模块复用范式
5. 通过测试验证 TraceNo 透传、响应头回写、日志脱敏和关键日志输出

**Rollback strategy**

- 若日志增强引起异常，可先回滚日志过滤器与前端 TraceNo 透传逻辑
- 本次变更不涉及数据库结构迁移，回滚风险主要集中在请求链路日志与测试

## Open Questions

- 当前无。首期默认日志输出到应用标准日志，后续如需接入集中式日志平台，再通过新的 OpenSpec change 扩展。
