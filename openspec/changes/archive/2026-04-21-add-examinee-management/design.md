## Context

当前仓库已经完成管理端登录、权限控制与 `TraceNo` 日志基础能力，但前端只有登录页和首页，后端也只有认证与首页接口。考生管理是路线图里的独立模块，依赖现有管理端认证和权限能力，需要同时新增 `exam-web` 的管理页面与 `exam-service` 的 REST、DTO、Mapper 和数据库设计，并明确登录成功后的管理端主页面要作为统一模块入口，既承载本次新增的考生管理，也承载后续其他管理模块，而不把考试计划或考生端登录混入本次 change。

## Goals / Non-Goals

**Goals:**
- 在 `exam-web` 中新增落在 `src/modules/examinees/` 的考生管理页面与业务服务层，支持查询、新增、编辑、删除、状态管理、批量导入导出。
- 明确登录成功后的管理端主页面是统一模块导航容器，至少能展示首页与考生管理入口，并保留后续模块接入方式。
- 在 `exam-service` 中新增 `cn.jack.exam.controller.admin`、`service`、`dto`、`mapper`、`entity` 分层下的考生管理实现，提供清晰的 REST 契约。
- 为考生基础信息建立可演进的 MySQL 表结构、MyBatis-Plus Mapper 与初始化/测试数据，支持后续模块复用。
- 复用现有权限、`TraceNo`、请求/响应/异常日志与脱敏规则，为考生管理增加可检索的关键业务日志。
- 明确本次 change 的 TDD 范围：`exam-service` 全量适用 TDD，`exam-web` 的接口封装、状态处理、表单校验与权限/路由逻辑适用 TDD。

**Non-Goals:**
- 不重做管理端登录、Token、角色权限基础设施。
- 不实现题库管理、考试计划、考生端登录、确认信息、在线答题或交卷流程。
- 不引入异步导入任务、消息队列或对象存储等超出当前模块的基础设施。
- 不把考生与未来考试计划的绑定规则提前实现到本次 change。

## Decisions

### Decision 1: 登录成功后的管理端主页面作为模块导航容器，`modules/examinees` 作为其中一个业务模块接入

`exam-web` 登录成功后进入的管理端主页面继续由统一布局承载，负责展示首页、考生管理以及后续其他管理模块的导航入口。`exam-web` 将在 `src/modules/examinees/` 下拆分 `pages`、`components`、`services`、`types`，并通过现有 `AdminLayout` 菜单挂接考生管理入口。考生管理模块内部使用一个列表页承载筛选、表格、分页、导入导出与新增/编辑弹窗或抽屉，避免把业务逻辑继续堆到 `App.tsx`。

选择这一方案是因为当前前端规模较小，但已经建立了 `modules/` 与 `shared/` 的分层约定；统一主页面可以稳定承载模块扩展，单页列表加弹层编辑可以覆盖完整的管理动作，同时保留后续拆分详情页的空间。备选方案是把每个模块都直接挂在 `App.tsx` 或为新增/编辑拆成独立路由页面，但前者会让首页容器失去扩展性，后者会在当前阶段引入更多路由和状态同步复杂度，而收益有限。

### Decision 2: REST 契约使用资源化的考生接口，并把权限控制细化到菜单和动作级别

后端统一暴露 `/api/admin/examinees` 资源，按能力拆分为分页查询、创建、更新、删除、状态更新、批量导入和条件导出接口。请求与响应通过 `dto/examinee` 下的显式 DTO 表达，不直接暴露实体。权限数据沿用现有 `admin_permission` 模型，至少增加一个考生管理菜单权限和对应的查询、创建、更新、删除、状态、导入、导出 API 权限码，并继续通过现有鉴权拦截器和 `@RequirePermission` 保护接口。

该方案与当前管理端资源命名和权限控制方式一致，也能让 `exam-web` 根据返回的菜单与权限决定页面入口和操作按钮可见性。备选方案是只配置一个粗粒度 `examinee:*` 权限，但这会降低受限管理员场景的可控性。

### Decision 3: `exam-service` 引入独立考生主数据表，并使用软删除保持后续扩展余地

后端新增 `examinee` 表，基础字段至少包含 `id`、`examinee_no`、`name`、`gender`、`id_card_no`、`phone`、`email`、`status`、`remark`、`deleted`、`created_at`、`updated_at`。`examinee_no` 作为稳定的业务编号并设置唯一约束，`id_card_no` 在有值时要求唯一，`status` 采用 `ENABLED`/`DISABLED` 这样的稳定枚举值。删除操作优先采用软删除，使后续考试计划等模块接入时不需要先推翻本次数据模型。

这一选择兼顾了当前 CRUD 需求与后续演进空间，并与 MyBatis-Plus 常见的逻辑删除实践相容。备选方案是物理删除记录，但后续如果考试计划引用考生，会增加迁移与审计成本。

后端代码放置建议如下：
- `cn.jack.exam.controller.admin`：考生管理 REST 接口
- `cn.jack.exam.service.examinee`：查询、维护、导入导出与状态管理流程
- `cn.jack.exam.dto.examinee`：查询条件、请求体、响应体、导入结果 DTO
- `cn.jack.exam.entity`：考生实体
- `cn.jack.exam.mapper` 与 `resources/mapper`：MyBatis-Plus Mapper 与 SQL
- `resources/db/mysql` 与测试库 schema/data：表结构、索引、权限种子与测试数据

### Decision 4: 批量导入导出统一采用 Excel 文件，导出列定义与导入模板保持一致

批量能力使用 Excel 兼容格式实现，优先采用单文件同步处理。导出接口按当前筛选条件输出固定列顺序的考生数据；导入接口接收相同列定义的文件，对每一行做必填项、格式、唯一性和状态值校验，并返回成功数、失败数以及逐行失败原因。实现上 `exam-service` 可引入成熟的 Excel 处理库，例如 Apache POI。

统一导入/导出列定义的好处是降低管理端培训成本，也避免为本次 change 再额外设计模板协议。备选方案是 CSV，但在中文管理场景下 Excel 兼容性和人工编辑体验更稳妥；异步导入任务则超出当前模块范围。

### Decision 5: 复用现有 TraceNo 与日志脱敏基础能力，并在考生管理中追加业务日志

所有考生管理接口继续复用现有 `TraceNo` 透传、请求/响应日志、异常日志和 `LogSanitizer` 脱敏规则。除此之外，`exam-service` 需要为新增、编辑、删除、状态变更、导入和导出输出关键业务日志，日志中仅记录脱敏后的考生标识摘要，不记录明文身份证号、完整手机号或导入原始敏感字段。

该方案遵守现有 OpenSpec 约束，也为后续联调和排障提供稳定链路。备选方案是只依赖通用 HTTP 日志，但那会让批量导入失败、状态切换等关键业务行为难以定位。

## Risks / Trade-offs

- [导入文件规模过大导致接口耗时上升] → 先限定同步导入的文件规模与行数，并在接口和 UI 中返回清晰失败提示，超大批次后续再拆分异步方案。
- [考生唯一字段定义过少会影响后续业务约束] → 本次先固定 `examinee_no` 为必填唯一键，`id_card_no` 为有值唯一，并在规格中显式记录基础字段范围。
- [软删除增加查询条件复杂度] → 通过 Mapper 默认过滤 `deleted = 0`，并在测试中覆盖列表、导出和重复校验场景。
- [前后端同时新增复杂表单，契约容易漂移] → 先以 OpenSpec 中的字段和接口定义为准，再用 `exam-service` 与 `exam-web` 的测试覆盖请求/响应结构。
- [导入错误信息过于技术化影响运营使用] → 返回行号、字段名与可读错误原因，前端直接展示摘要，不暴露内部异常栈。

## Migration Plan

1. 在 `exam-service` 新增数据库迁移脚本，创建考生表、索引和菜单/API 权限种子数据，并同步更新 H2 测试 schema/data。
2. 先完成 `exam-service` 的测试、DTO、Mapper、Service、Controller 与导入导出实现，再提供可联调的 REST 接口。
3. 在 `exam-web` 接入考生管理模块、菜单与路由，并保证登录成功后的主页面可统一承载首页、考生管理和后续模块入口，再基于接口完成页面联调。
4. 联调阶段验证 `TraceNo` 透传、日志脱敏、权限受限账号行为、导入失败提示和导出文件结构。
5. 回滚策略以“禁用菜单/权限入口并回退应用版本”为主；若数据库尚未被后续模块引用，可通过单独迁移脚本下线考生表与权限数据。

## Open Questions

当前没有阻塞 proposal 落地的开放问题。本次默认的考生基础字段集为 `examineeNo`、姓名、性别、身份证号、手机号、邮箱、状态、备注；如后续需要学校、院系、证件照片等扩展字段，应通过新的 OpenSpec change 增补，而不是在本次 change 中扩边界。登录成功后的主页面在本次只要求建立统一模块容器与考生管理入口，不在本次 change 中定义后续模块的具体业务行为。
